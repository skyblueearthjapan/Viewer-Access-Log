# Viewer-Access-Log 設計ドキュメント

> 技術部フォルダへのアクセスを「ビューアー経由」「サーバー直接」「未帰属」で**明確に区別**し、
> NTFSロックダウンが「ユーザーをビューアーに矯正できているか」を**効果測定**する、
> **完全分離・読み取り専用**のダッシュボード。

本書は Claude（設計）と OpenAI Codex（独立レビュー, gpt-5.5）の意見を統合した確定方針。

---

## 1. 背景と既存システム（壊さない）

| システム | 役割 | 本アプリとの関係 |
|---|---|---|
| **Secure-File-Explorer (SFE)** | 代理アクセス方式のファイルビューアー。OpenFile/Search を **SQLite + 月次Excel** に記録 | 🟦 **ビューアー経由ログ**の取得元（読み取りのみ） |
| **AuditLogger** | Windows監査イベント(4663等)を PostgreSQL16 へ全量収集。Blazor UI。本番稼働中だが重い | 🟥 **直接アクセスログ**の取得元（読み取りのみ） |

**鉄則**：既存2システムは**一切変更しない**。本アプリは両者を **SELECT専用で読むだけ**（保険）。

---

## 2. 中核思想：2色ではなく「3色」

Codexレビューでの最重要修正。**「audit_logs の `MTSV$`/NULL を除外した＝ビューアー経由」と断定するのは誤り**
（同じマシンアカウントには他サービス・バックアップ・AV・手動プロセスも混じる。NULLはSID解決失敗等）。

正しい分類は **3バケット**：

| 色 | 区分 | 判定根拠（確実性） |
|---|---|---|
| 🟦 **青 = ビューアー経由** | SFE SQLite に存在する操作 | **本人特定・確実**（アプリ自身の記録） |
| 🟥 **赤 = サーバー直接** | audit_logs で `LINEWORKS-NET\実ユーザー` ＋ `is_content_read=TRUE` ＋ 対象パス配下 ＋ サービス/マシンアカウント除外済み | **強く"直接アクセス"と言える** |
| ⬜ **灰 = 未帰属** | `MTSV$` / `NULL` / サービスアカウント / SID不明 | **ビューアーの証明ではない**。可視化して誤った安心感を防ぐ |

> 青は SFE で確定、赤は audit で検出、灰は audit の残りを「未帰属」として**隠さず見せる**。

## 3. アーキテクチャ（完全分離・読み取り専用の "投影" アプリ）

```
        SELECT専用                         SELECT専用(mode=ro)
AuditLogger PostgreSQL ─┐             ┌─ SFE SQLite
 (viewerロール・接続1-2・ │             │  (技術部のOpenFile/Search)
  statement_timeout)     ▼             ▼
                  ┌──────────────────────────┐
                  │  Sync Worker(定期同期)    │  ← UIからソースDBを直接叩かない
                  │  技術部スライスのみ正規化   │
                  └──────────┬───────────────┘
                             ▼
                  ┌──────────────────────────┐
                  │  自前キャッシュ(SQLite)    │  raw / normalized / rollup / sync_health
                  └──────────┬───────────────┘
                             ▼
                  ┌──────────────────────────┐
                  │ Minimal API + 軽量UI(HTML) │  ← キャッシュだけ参照=常に高速・落ちない
                  │ 3色 + GAP健全性を同時表示  │
                  └──────────────────────────┘
```

**設計判断（Claude×Codex 一致）**
- ライブ直結 ✗ ／ 既存DB相乗り ✗ → **定期同期キャッシュ ◯**（重さ・障害に巻き込まれない）。
- UI = **軽量HTML + Minimal API**（Blazor Serverはcircuit/状態で重くなりやすく、1画面用途には過剰）。
- キャッシュ = **専用SQLite**（技術部限定・少人数・保持数ヶ月〜1年で十分。多部門展開時は専用PostgreSQLへ）。

## 4. データソース別の取得ルール

### 4-1. 🟦 青（SFE SQLite）
- SFE のアクセスログ（`OpenFile` / `Search` 等）を読む。**これだけが「ビューアー経由」として確実**。
- 読み取りは `mode=ro`、ロック時は短時間リトライ（必要ならスナップショットコピーを読む）。

### 4-2. 🟥 赤（AuditLogger audit_logs）狭いクエリ
```sql
SELECT id, event_time, user_name, action::text, file_path, folder_path, file_name,
       process_name, host(source_ip) AS source_ip, result::text
FROM   audit.audit_logs
WHERE  event_time >= @from AND event_time < @to          -- 月次パーティションを時刻で必ず絞る
  AND  folder_path ~* '[\\/]Data[\\/]技術部([\\/]|$)'     -- 対象パス
  AND  user_name ~ '^LINEWORKS-NET\\'                     -- 実ユーザー
  AND  user_name !~* 'MTSV\$'                             -- マシンアカウント除外
  AND  user_name !~* '(NETWORK SERVICE|LOCAL SYSTEM|SYSTEM|svc[_-])'  -- サービス除外
  AND  is_content_read = TRUE                             -- 中身読取のみ(属性/一覧の副産物を除外)
  AND  result::text = 'Success'
ORDER BY event_time DESC;
```

### 4-3. ⬜ 灰（未帰属）
- 上記で除外した `MTSV$` / `NULL` / サービスを**捨てずに**「未帰属」として件数集計・可視化。

## 5. 二重カウント・二義性の対策（Codex指摘）

- **生イベントをそのまま数えない**。同一ユーザー×同一ファイル×近接時刻×同一セッション×同種操作は **1アクセスに畳む**。
- **SFE の OpenFile** と **audit の MTSV$ 読取** は同一実体 → **二重表示しない**（青で確定済みの実体は灰でも重複させない）。
- `FOLDER_LIST` / `FILE_METADATA_READ` / `FILE_PREVIEW` / `FILE_OPEN` は**別カテゴリ**。フォルダ列挙は内容読取から分離。
- `4663` の多発を「複数ファイル持ち出し」と短絡しない。
- **パス正規化**：UNC/ローカル/DFS/大小文字/Unicode/リネームを正規化してから突き合わせる。

## 6. 指標（生ログ一覧でなく「効果測定ダッシュボード」）

- 🟦 ビューアー経由 Open 数
- 🟥 直接アクセスした実ユーザー数 / ファイル数
- ⬜ 灰（マシン/不明）件数
- ユーザー別 / 時間帯別の 🟥 件数
- **監査GAP時間帯**（AuditLoggerが止まっていた区間）
- **ビューアー利用率 = 青 / (青+赤)** … ただし **GAP時間帯は分母から除外**
  （止まっていた時間を「直接ゼロ」と表示すると経営判断を誤らせる — Codex）

生ログへのドリルダウンは残すが、**通常画面は集計中心**。

---

## 7. 安全性（既存を壊さない＝保険）

- 既存DBへ **書き込み一切なし**。PostgreSQL は **`viewer` ロール（GRANT SELECT のみ）**。
- 接続数 1〜2、`statement_timeout` 付き。SFE SQLite は `mode=ro`。
- 失敗しても既存は無傷。新アプリは捨てるだけ。

## 9. 将来の改善（今すぐではない・記録）

- **SFEを NetworkService → 専用 gMSA** で動かす → `MTSV$` より遥かに判別しやすい。
- SFEに **リクエストID** を持たせ、SFEログと監査ログを時間/パス/IDで相関。
- 究極は **構造的統制**：技術部はNTFSで実ユーザーから読めなくし、直接アクセスを「検知」でなく「構造的に失敗」させる（＝現在進行中のロックダウン）。本アプリはその残党と未帰属を可視化する位置づけ。
